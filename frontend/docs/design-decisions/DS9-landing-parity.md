# DS9 — Landing ↔ App token parity

> **Decision:** `landing/` consumes the same `colorTokens.cjs` as the
> RN app. Brand color, pastel performers, and warm canvas are sourced
> once. A waitlist signup → first app launch is now visually one
> product.

## Why

A user lands on `sazonchef.com`, sees coral + warm cream + Fraunces
hero, fills out the waitlist, downloads the app. If the app's coral
is a different coral, that's a brand fracture they read instantly
(even if they can't articulate it). DS9 closes that gap.

Before DS9:
- Landing brand: `#E8704F` (warmer, more red-shifted)
- App brand: `#fa7e12` (more saturated orange)

These are both "coral" but visually distinct. After DS9:
- Both use `#fa7e12` from `frontend/constants/colorTokens.cjs`.

## What ships

### DS9.1 — `landing/tailwind.config.ts` consumes the shared source

```ts
const t = require('../constants/colorTokens.cjs');
const base = t.Brand.light.base;  // canonical brand
const deep = t.Brand.light.deep;  // canonical brand-deep
```

The 50–900 numerical scale is generated by mixing `base` toward white
(50–400) and toward `deep` (600–900), so existing landing utilities
(`bg-brand-100`, `text-brand-700`) keep working without rewriting
landing components.

### DS9.2 — Pastel + Surface tokens mirror the app

| Landing token | Source |
|---------------|--------|
| `pastel.{peach,sage,lavender,sky,blush,golden}` | `t.PastelTokens.light.*` |
| `surface.cream` | `t.Canvas.warmLight` (DS2.2 — Today's editorial canvas) |
| `surface.ink` | `t.Ink.light.warm` |
| `surface.linen` / `surface.smoke` | landing-specific (no 1:1 in app) |
| `accent.{paprika,basil,saffron}` | landing-specific (marketing only) |

### DS9.3 — Visual continuity

The landing → app journey now shares:

- **Same coral.** `Brand.light.base = #fa7e12` everywhere.
- **Same warm cream canvas.** `Canvas.warmLight = #FAF7F4`.
- **Same pastel performers.** Sage, golden, lavender, peach, sky,
  blush — identical hex values.
- **Same Fraunces display family.** Already aligned via Google Fonts
  (frontend uses `@expo-google-fonts/fraunces`, landing uses Next's
  `next/font/google` with `Fraunces`).

## Visual change to expect on the landing site

The brand coral shifts from a warmer, more muted tone (`#E8704F`) to a
brighter, more saturated orange (`#fa7e12`). On the marketing hero,
this reads as more energetic / appetite-forward — closer to the
brand's "real food, made for this user" voice than the previous
restaurant-warm coral.

If the new coral feels harsh in any specific landing surface, the fix
is to source from `Brand.light.deep` (`#d67a0c`) instead — that's the
muted variant the frontend ships for emphasis states.

## Anti-patterns

- Hardcoding `#fa7e12` (or any token's value) in landing JSX. Always
  reach for the named utility (`bg-brand-500`).
- Mirroring tokens manually on the landing side without importing
  from the shared source — recreates the dual-source problem.
- Adding a landing-only "brand-marketing" coral that diverges from
  the app brand. If marketing needs more variety, add a *named*
  marketing accent (we already have `accent.paprika / basil /
  saffron`).

## Test enforcement

`frontend/__tests__/design/landingParity.test.ts` (6/6):
- Landing config imports from `../constants/colorTokens.cjs`.
- No `#fa7e12` literal outside that import.
- `brand.500 === Brand.light.base`.
- Every pastel key maps to the canonical pastel token.
- `surface.cream === Canvas.warmLight` (DS2.2 alignment).

## Source of truth

`frontend/constants/colorTokens.cjs` is the canonical source.
`frontend/landing/tailwind.config.ts` is the consumer.
