## 🎯 The Thesis

Sazon is **the lifestyle app for the eater who's past the spreadsheet.**

The user has graduated from optimization (cut/bulk/maintain) to **enjoyment + nourishment + cultural curiosity.** They want delicious whole-food meals from around the world, with the engine quietly making every recommendation sharper. They track macros AND micros — not obsessively, but as a discovery surface that fits how they want to eat.

Three rules — louder than any north star we've shipped:

1. **The bottom bar is a journey, not an object catalog.** Tabs answer *"what am I doing?"* not *"what kind of thing is this?"* See `plans/ia-spec.md`.
2. **Personalization without measurement is theater.** Every recommendation surface emits a signal-yield metric; surfaces with low yield are noise and get cut.
3. **Generic content is the failure mode.** A static "Trending now" feed shown to two users at the same time is a bug, not a feature.

For every entry below, the spec must answer:
- *What new signal does this collect about the user?*
- *Which existing recommendation does this make smarter?*
- *Is this discovery (invitation) or optimization (correction)?* — Default to discovery.
- *What's the kill criteria — when do we cut it?*

## 🍽️ Tier D — Recipe DB & Content Quality (gates v1 launch)

> **Reframed 2026-05-05:** the original Tier D chased raw recipe count (4,735+ across 134 cuisines). With Tier T-bis's retrieval + LLM ranker doing the heavy lifting, **a curated 1,500–2,500-recipe catalog with deliberate cuisine breadth outperforms a sprawling one with mediocre coverage.** Quality > quantity, but breadth (cuisines from around the world) is non-negotiable — it's the lifestyle-eater positioning.
>
> **New principles:**
> 1. **Content-only quality scoring.** Engagement signals (views, cooks, saves) are biased toward cuisines we already surface — using them as a quality proxy would tautologically narrow the catalog. Score on content properties only.
> 2. **Coverage matrix protects breadth.** Every cuisine must satisfy a minimum dish-archetype matrix (weeknight main, weekend project, comfort/stew, etc.). A "delete" never shrinks coverage — if a low-quality recipe is the only one filling an archetype slot, we *rebuild* it before deleting.
> 3. **Underrepresented cuisines are a moat.** Filipino, Eritrean, Ethiopian, Georgian, Levantine, Senegalese, Peruvian, Trinidadian, Lao, Burmese, Oaxacan, Yucatecan, Sicilian, Sardinian, Andalusian, Hungarian — these get *deliberate* acquisition, not opportunistic.
> 4. **Two-phase triage.** Score everything first → triage with judgment → never delete blindly.

## 🔁 Tier M — Self-Improvement Engine (meta / product-management automation)

#### Tier M guardrails (non-negotiable)

- **Persona is the gate, not the suggestion.** A statistically-popular pattern that violates `persona.md` never ships, regardless of engine confidence.
- **No autonomous shipping.** Proposals require human review. Always.
- **No PII in observations.** Coach feed stores intent meta only — never message body.
- **No silent prompt promotion.** Quarterly audit drafts v(N+1); human must promote.
- **10 proposals/week max.** If the engine wants 30, the prompt is broken.
- **Kill switch always works.** `SELF_IMPROVEMENT_ENGINE_ENABLED=false` short-circuits every routine.

## 🧪 Tier T-bis — Recommender Bootstrap (the brain behind Tonight)

> **Thesis:** Tonight mode lives or dies on first-try accuracy. Today's 70/30 macro/taste blend was built for a *feed* (rank 50 candidates, user picks one) — not a *proposal* (output the single right answer). Rebuild the recommender as a three-stage pipeline: cheap retrieval (item embeddings) → LLM ranker (during cold start) → distilled specialized model (once we have ≥50k real interactions). Each stage is independently shippable; together they take Sazon from rule-based ranking to a learning system.
>
> **Dependency on Tier T:** T1.1 (`/api/tonight/proposal`) reuses TB1's retrieval + TB2's ranker. T-bis must ship at least through TB2 before T's proposal endpoint can feel mind-reading. TB0/TB1 can also lift the *current* 4-tab IA's home feed quality, so this work pays for itself even if Tonight doesn't ship.
>
> **Honest tradeoffs to name up front:**
> - TB2 (LLM ranker) makes Sazon partially dependent on Claude API for ~6 months. Mitigated by fallback to current rule-based ranker on API failure.
> - TB0 (Food.com embeddings) trains on *other people's* interaction data — the item latent space transfers, the user latent space does not. We use it for retrieval only, not for user modeling.
> - We are *not* training on synthetic users (LLM-generated personas) as primary signal. Synthetic data appears here only as test fixtures (TB4). Real labels come from TB3 logging.

- [x] **TB1.1: User-context vector builder** ✅ — `services/recommender/contextVector.ts` averages embeddings of the user's last `maxCookHistory=30` cooks with exponential time decay (`halfLifeDays=21`), then adds an L2-normalized pantry-bias term sourced from `LeftoverInventory` rows whose `expiresAt >= asOf`. Returns `{ vector, cookCount, pantryBoostCount }`; vector is null on cold start with no leftovers.
  - **Test:** `backend/__tests__/recommender/contextVector.test.ts` ✅ 5/5 — null on no-cook+no-leftover; cooks-centroid > random-cuisine sim; recent > old via half-life decay; pantry bias measurable shift; `take=maxCookHistory` query cap.
- [x] **TB1.2: Top-K retrieval** ✅ — `services/recommender/retrieveCandidates.ts`. Hard filters first (allergens by ingredient-substring match, `maxCookTime`, pantry coverage threshold), then cosine similarity over the survivors; deterministic order ties broken by recipeId. SQLite-friendly in-memory loop, no vector DB.
  - **Test:** `backend/__tests__/recommender/retrieveCandidates.test.ts` ✅ 7/7 — top-K by cosine; allergen + cook-time filters; <K survivors when filters thin the pool; deterministic across runs; **<100ms on a 1500-recipe synthetic catalog**; null-embedding rows skipped.

  - **Test:** `backend/__tests__/services/recommender/homeFeedRetrievalAdapter.test.ts` ✅ 4/4 — flag-off skip; cold-start returns null; happy-path top-K; thrown error → null (caller falls back).

### TB2: LLM-as-ranker (cold-start policy)

> **Goal:** given top-50 candidates + full user context, output the single best pick + a one-line reason. Same call generates the Tonight copy line — no separate copy service needed.

- [x] **TB2.1: `recommenderService.rankWithLLM`** ✅ — `services/recommender/recommenderService.ts`. Sub-threshold confidence returns `null` for graceful UI fallback.
  - **Test:** `backend/__tests__/services/recommenderService.test.ts` ✅ 8/8 — valid-shape parse; reason truncate; invalid JSON / thrown error → fallback; below-threshold null; phantom recipeId → fallback; empty candidates → null; selfRating→confidence map.
- [x] **TB2.2: Wire ranker into `/api/tonight/proposal`** ✅ — `modules/tonight/{tonightRoutes,tonightController}.ts` mounted at `/api/tonight` behind `authenticateToken`.
